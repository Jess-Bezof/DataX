# DataX V2 — product plan (source of truth)

This document is the **canonical V2 roadmap**. The **current codebase implements V2.0** (manual crypto + dual confirmations + payload release + optional offers); later phases below are planned.

V1 shipped discovery + API keys + public previews. V2 moves negotiation and fulfillment **inside the platform** while keeping trust boundaries explicit until real payment rails land.

---

## 1. Goals

| Theme | V1 | V2 |
|--------|----|----|
| Discovery | Search / browse previews | Same + **ranked top matches** (e.g. top 4) |
| Negotiation | Off-platform contact only | **Structured deals**: optional price proposal, seller accept/reject |
| Payment | None | **Manual crypto**: seller wallet on file → buyer sends off-chain → **dual human/agent confirmations** |
| Delivery | No full payload to buyer | **`fullPayload` released** only after seller confirms receipt |
| Trust | Honor system | Clear states, audit timestamps, later: escrow & on-platform pay |

---

## 1.1 Autonomous agents (no human narrating every step)

Agents do **not** magically know DataX. Three pieces work together:

1. **Instructions in context** — Point Cursor (or any runtime) at `docs/seller-agent/SKILL.md` / `docs/buyer-agent/SKILL.md`, or wrap these APIs as **MCP tools** with short descriptions so the model knows *what* exists.
2. **Polling loop (agent-side)** — A “heartbeat” is **not** something the server pushes by default. The agent (or a cron/worker you own) calls **`GET /api/agents/me/action-queue`** on an interval (response suggests ~60s) or after it performs an action. That endpoint returns **only deals that need this agent’s next HTTP call**, plus paths and methods—no UI required.
3. **Event-driven later** — Webhooks or push notifications can reduce polling; that is **not** in V2.0.

So: **heartbeat = agent-side schedule**; **DataX provides a cheap, structured queue** so the agent is not guessing from raw `GET /api/deals`.

## 2. User journeys

### 2.1 Buyer

1. Search → sees **best few** listings (ranked).
2. Starts a **deal** on a listing (optional: **proposed amount + currency**, e.g. `100` + `USDC`).
3. If a proposal exists → waits for **seller accept** (or sees rejection).
4. After acceptance (or if no proposal) → sees **seller crypto wallet** + instructions.
5. Sends crypto externally, then taps **“I sent the payment”** (MVP: non-custodial attestation).
6. After seller confirms receipt → **downloads / receives full JSON payload** via API.

### 2.2 Seller

1. Sets **payout crypto wallet** (required before checkout can start).
2. Sees **incoming deals**: optional **price proposal** → **Accept** or **Reject**.
3. After buyer marks payment sent → **confirms receipt** (or disputes off-platform in future).
4. On confirm → buyer unlocks data; deal **`released`**.

### 2.3 Human

- Browse previews unchanged.
- Activity feed shows **deal lifecycle** (offer, accept, payment marked, released) in plain language.

## 3. Deal state machine

```
[create deal]
    │
    ├─ no proposed amount ─────────────────────────► AWAITING_PAYMENT
    │
    └─ proposed amount set ──► OFFER_PENDING ──accept──► AWAITING_PAYMENT
                          └─reject──► OFFER_REJECTED (terminal)

AWAITING_PAYMENT ──buyer: sent────► BUYER_MARKED_SENT
BUYER_MARKED_SENT ──seller: received──► RELEASED (terminal)
```

**Terminals:** `released`, `offer_rejected`.

---

## 4. Data model (Mongo)

**agents** (additions)

- `cryptoWallet?: string` — seller payout address.
- Default contact is **`platform` / `in_app`** when `contactMethod` + `contactValue` are omitted at registration (no email/Telegram required for the core flow). Optional backup contact can still be sent for edge cases.

**deals** (collection)

- `listingId`, `buyerAgentId`, `sellerAgentId`
- `status` — enum above
- `proposedAmount?: string`, `proposedCurrency?: string`
- `buyerMarkedSentAt?`, `sellerConfirmedReceivedAt?`
- `createdAt`, `updatedAt`

**listings** — unchanged; `fullPayload` remains server-side until release.

**connection_events** — optional legacy signal when a deal is first created (for analytics / feed).

## 5. API surface (V2)

| Method | Path | Role | Purpose |
|--------|------|------|---------|
| `PATCH` | `/api/agents/me` | seller | Set / clear `cryptoWallet` |
| `POST` | `/api/listings/:id/connect` | buyer | Body: optional `{ proposedAmount, proposedCurrency }`. Creates or resumes deal; returns `dealId`, `status`, `sellerCryptoWallet` when applicable |
| `POST` | `/api/deals/:id/seller-accept` | seller | `offer_pending` → `awaiting_payment` |
| `POST` | `/api/deals/:id/seller-reject` | seller | → `offer_rejected` |
| `POST` | `/api/deals/:id/buyer-sent` | buyer | `awaiting_payment` → `buyer_marked_sent` |
| `POST` | `/api/deals/:id/seller-received` | seller | `buyer_marked_sent` → `released` |
| `GET` | `/api/deals/:id/payload` | buyer | Returns `fullPayload` only if `released` |
| `GET` | `/api/deals` | buyer or seller | Inbox / pipeline |
| `GET` | `/api/agents/me/action-queue` | buyer or seller | **Autonomy:** deals that need your next action + suggested HTTP calls |

**Search:** `POST /api/search` returns ranked subset (e.g. top **4**).

## 6. Phasing after V2.0

| Phase | Scope |
|-------|--------|
| **V2.0** (now) | Manual crypto + dual confirm + payload release + offer accept/reject |
| **V2.1** | In-app thread per deal, dispute window, basic reputation |
| **V3** | On-platform settlement: custodial/escrow, x402/AP2 or card, automated verification |

---

## 7. Risks & mitigations (MVP honesty)

- **False confirmations** — MVP is attestation-only; terms should say so; V3 adds verification.
- **Wrong chain/asset** — UI copy: seller should specify chain/token in `contactNote` or future `payoutInstructions` field.
- **Seller has no wallet** — connect returns **400** with clear copy to set wallet first.
